🪜 2. REST o Eventos

Interacción REST Evento ¿Por qué?
1 X La busqueda de disponibilidad se limita a la visualizacion de la disponibilidad y no desencadena acciones adicionales,por eso, la comunicacion entre estos dos servicios es de forma sincorna, lo que garantiza que el usuario reciba en tiempo real la información solicitada sin esperas innecesarias.
2 X Consideramos que es evento por que no es una consulta directa, sino una reaccion del sistema a algo que ocurrio, en este caso la creación de cita. El historial solo necesita enterarse del evento y registrar los datos, sin interrumpir el proceso principal que es el agendamiento.
3 X Es envento porque el servicio de historial solo informa que ocurrio un cambio y el servicio de notificacion reacciona a ese evento, enviado al usuario correspondiente.

🪜 3. Síncrono o Asíncrono

Interacción Síncrono Asíncrono Justificación
1 X Porque el usuario requiere ver los horarios disponibles inmediatamente tras la consulta, no se necesita que se hagan mas acciones que mostrar el resultado de dicha busqueda
2 X Porque la crecion de cita no debe esperar a que el historial la registre. El usuario confirma la cita y en segundo plano el historial se actualiza.
3 X El envio de notificaciones no debe bloquear el sistema, el evento es escuchado y el servicio de notificaciones lo procesa en segundo plano

🪜 5. Análisis de Fallos

Gestion de Citas
Si el servicio de gestión de citas falla, los pacientes no podrán agendar, cancelar ni cambiar sus citas médicas, y los doctores no podrán visualizar los pacientes agendados. Esto afectaría directamente la atención médica, generaría desorden en los horarios y frustración en los usuarios, ya que el sistema dejaría de cumplir su función principal.
Si el servicio de gestión de citas falla, el sistema debería continuar funcionando de manera parcial, permitiendo el acceso de los usuarios y la consulta de citas agendadas en el servicio de historial, pero sin la posibilidad de crear, modificar o cancelar citas hasta que el servicio sea restablecido.
Si falla el servicio de gestión de citas, el sistema puede manejarlo mostrando un mensaje claro al usuario indicando que el agendamiento está temporalmente inhabilitado, mientras permite usar otras funciones como iniciar sesión y consultar información disponible. Además, se puede usar un “circuit breaker” para evitar seguir llamando al servicio cuando está fallando y así no empeorar el problema

🪜 6. Service Discovery

Si la dirección IP de un servicio cambia, los demás servicios no podrán encontrarlo ni comunicarse con él, porque seguirán intentando conectarse a la dirección anterior. Como resultado, algunas funciones del sistema dejarían de funcionar temporalmente, mostrando errores o tiempos de espera prolongados, hasta que se actualice la nueva dirección o se restablezca la conexión correctamente.
Un servicio puede saber dónde encontrar a otro si la dirección está configurada previamente en el sistema (por ejemplo, guardada en la configuración). Sin embargo, si la IP cambia y no se actualiza esa información, los servicios no podrán comunicarse correctamente.
Service Discovery sería útil porque permite que los servicios se encuentren automáticamente sin depender de una IP fija. En lugar de guardar una dirección específica, los servicios consultan un registro que les indica dónde está disponible el otro servicio en ese momento. Así, si la IP cambia, el sistema sigue funcionando sin necesidad de hacer cambios manuales.
Lo aplicaríamos en la comunicación entre los servicios principales, especialmente entre el servicio de gestión de citas, autenticación y notificaciones. De esta manera, si alguno cambia de dirección o se mueve a otro servidor, los demás servicios podrán ubicarlo automáticamente y continuar funcionando sin interrupciones.
